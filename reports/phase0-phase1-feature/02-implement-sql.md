## Design Notes

### users table
- No `password_hash` — credentials managed entirely by BetterAuth
- No `updated_at` — user profile mutations not in Phase 1 scope
- `UpsertUser` uses `ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name` — safe for first-login upsert; email is stable, name may change in BetterAuth profile

### org_memberships table
- `UNIQUE (org_id, user_id)` constraint enforces one membership row per user per org
- FK ON DELETE CASCADE on both `org_id` and `user_id` — membership is cleaned up automatically when org or user is deleted
- Queries filter by `user_id` (multi-org user listing) or by `(user_id, org_id)` pair (single membership lookup)
- No `org_id`-only listing query — org member roster is a Phase 2 feature (not required by any Phase 1 endpoint)

### org_invites table
- `accepted_at TIMESTAMPTZ` nullable — NULL means pending, non-NULL means accepted
- `token TEXT NOT NULL UNIQUE` — opaque 32-byte base64url value generated by service layer
- `expires_at TIMESTAMPTZ NOT NULL` — enforced at service layer (7 days from creation); DB stores the value for auditability
- Index on `token` for O(1) lookup in `GetOrgInviteByToken`
- `GetOrgInviteByToken` returns full row including `accepted_at` — service checks nullability to detect already-accepted invites (Phase 2 flow)

### Multi-tenancy compliance
- `users` — top-level resource, no `org_id` column. Queries filter by `id` or `email` (sub claim from JWT)
- `org_memberships` — sub-resource; all queries filter by `user_id` or `(user_id, org_id)`. No unscoped listing.
- `org_invites` — sub-resource; `InsertOrgInvite` always includes `org_id`; `GetOrgInviteByToken` is token-scoped (invite token is unguessable, 32-byte random — acceptable for lookup without additional org filter)
