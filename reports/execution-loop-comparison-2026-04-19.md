### Качество выполненного (false positives)

Все 4 закрытых issue содержали конкретные требования по scope — opencode выполнил их формально (коммит + `gh issue close`), но не полностью:

| Issue          | Требование                                                   | Что сделал opencode                                 | Что сделал Claude Code                             |
|----------------|--------------------------------------------------------------|-----------------------------------------------------|----------------------------------------------------|
| #13 httputil   | `Response, ErrorBody, RespondJSON, RespondError`             | 4 символа, 8 вставок — **полностью**                | 4 символа, 4 вставки                               |
| #14 model      | «Все экспортные типы, константы, sentinel errors (8 файлов)» | 1 строка на файл → **~1 из N типов** на файл        | 61 вставка, все типы + константы + sentinel errors |
| #15 repository | «Все interfaces, methods, constructors (8 файлов)»           | 1 строка на файл → только первый символ (интерфейс) | 46 вставок, интерфейсы + методы + конструкторы     |
| #16 handler    | «9 NewXxxHandler functions»                                  | 9 godoc-строк — **полностью**                       | 9 godoc-строк                                      |

Вывод: **2 из 4 issues (#14, #15) закрыты формально, но фактически недоделаны** — модель добавила минимальный комментарий к первому встреченному символу и остановилась. Реальный процент выполнения = **2 / 16 = 12.5%** (если считать по полному соответствию требованиям issue), либо **4 / 16 = 25%** (если считать по `gh issue close`).

### Почему loop остановился

Qwen3 30B-A3B **галлюцинировала про пустой backlog**: после закрытия 4-й issue модель решила что весь пул задач выполнен, хотя `gh issue list --state open` показывает 12 открытых. Протокол execution-loop (перечитывать backlog после каждого коммита) был проигнорирован — модель опиралась на собственную память диалога, а не на команду `gh`.

Это принципиальное отличие от Claude Code, который каждый цикл начинает с re-fetch issues.

## Паттерн деградации локальной модели

```
Setup (AGENTS.md + subagents)    ── работает
Issue 1 (godoc handler)          ── pass (formal close)
Issue 2 (godoc repository)       ── pass (но scope недопокрыт)
Issue 3 (godoc model)            ── pass (но scope недопокрыт)
Issue 4 (godoc httputil)         ── pass (полностью)
Issue 5+ (bugs, tests, features) ── НЕ ПРОБОВАЛ: loop остановлен галлюцинацией
```

Корневые проблемы Qwen3 30B-A3B в режиме execution-loop:

1. **Игнорирует `gh issue list`** — после 4 коммитов решает что backlog пуст, не делая повторного запроса
2. **Минимальное scope-compliance** — добавляет годок к первому символу и считает issue закрытым, не вычитывая требования issue целиком
3. **Не использует батчинг** — коммитит каждый issue отдельно, хотя все 4 godoc-issues можно было решить одним коммитом (Claude Code так и сделал — `51cd253`)
4. **Собственная метрика** — пишет отчёт "100% success, 4/4" не сверившись с исходным backlog'ом

## Выводы

1. **Cloud (Claude Code Opus)**: 16/16 issues, 28 минут, 100% scope-compliance, 10 коммитов с батчингом связанных задач — полностью автономный execution loop без вмешательства
2. **Local (OpenCode + Qwen3 30B)**: 4/16 issues (фактически 2/16 по полному scope), 6.5 минут — закрыл только самый простой профиль (docs), и даже из него 2 issue недодокументировал. Остановился на галлюцинации "backlog пуст"
3. **Разрыв**: **×4** по количеству закрытых issues, **×8** по scope-compliance. При этом локальная модель выглядит быстрее в пересчёте на задачу (1.6 vs 1.75 мин) — только потому что брала самые простые задачи и делала их поверхностно
4. **Ключевая проблема Qwen3 в execution-loop**: отсутствие дисциплины перечитывания backlog. Протокол был прописан в AGENTS.md, subagents добавлены, permissions auto-approved — но модель доверилась собственной памяти вместо `gh issue list`, и объявила работу завершённой при 12 открытых issues
5. **False positives опасны**: issues #14 и #15 помечены closed, но требования не выполнены. Без ручной проверки это создаёт техдолг
6. **Оптимальная связка**: Claude Code для execution-loop и multi-file задач + локальная модель для автокомплита, быстрых подсказок и узких однострочных правок с явным scope
