### Почему сломалась на задаче #3

Задача #12 (Creative Update) требовала multi-layer генерации: sqlc query → repository → service → handler → router.

**Проблемы:**
1. **Не использует sqlc** — написала raw SQL через pgx напрямую вместо добавления query в `creatives.sql` и запуска `sqlc generate`. Прямое нарушение правил проекта
2. **Неправильная структура**: создала `internal/repo/creative_update.go` вместо добавления метода в существующий `internal/repository/creative.go`
3. **Потеря контекста**: не прочитала существующий код — продублировала `UpdateCreativeParams` с другими полями, несовместимыми с БД-схемой
4. **Build fail**: `go build` упал на несовместимых типах (pgtype vs uuid), модель зациклилась на фиксах — 3 итерации, каждая ломала что-то новое
5. **Остановлена вручную** после 3-й неудачной итерации фикса

**Корневая причина**: модель не способна удерживать контекст проекта (структуру папок, naming, sqlc pipeline) при генерации кода в 5+ файлов. Системный промпт в `.continuerc.yaml` слишком сжат и частично игнорируется.

## Паттерн деградации локальной модели

```
Задача 1 (простой рефакторинг) ── pass
Задача 2 (простой рефакторинг) ── pass (со 2-й попытки)
Задача 3 (multi-file feature)  ── fail, зацикливание
```

Локальная модель справляется с задачами, где:
- Изменение в 1-2 файлах
- Не нужно понимать архитектуру проекта
- Паттерн очевиден из самого файла

Ломается, когда:
- Нужно создать/изменить 3+ файлов
- Нужно следовать pipeline (sqlc → codegen → repository)
- Нужно интегрироваться в существующую структуру

## Выводы

1. **Cloud (Claude Code)**: 16/16 задач, 28 минут, 0 ошибок — полностью автономный execution loop
