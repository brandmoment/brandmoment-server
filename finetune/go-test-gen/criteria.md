# Критерии оценки «стало лучше»

Задача файнтюна — научить `gemini-1.5-flash-001` генерировать unit-тесты в стиле репозитория BrandMoment. Модель считается **успешно дообученной**, если на 10 eval-примерах она показывает улучшение по всем блокам ниже.

## Блок 1. Формальная корректность (чек-лист, 5 пунктов)

По каждому из 10 eval-ответов проверяем:

1. **Компилируется**: код можно вставить в соответствующий `*_test.go` и `go build ./...` проходит (после `goimports -w`).
2. **Верный нейминг**: имя теста совпадает с шаблоном `TestTypeName_Method[_Scenario]` для методов или `TestFuncName` для свободных функций.
3. **Без посторонних блоков**: только `func TestXxx(...) {...}` — нет `package`, нет `import`, нет `main`, нет пояснений текстом.
4. **Использует проектные хелперы**: для DB-error кейсов — `mockDBTX`, `errRow`, `errSentinel` из `mock_dbtx_test.go`; для UUID ↔ pgtype — `uuidToPgtype`, `pgtypeToUUID`.
5. **Покрывает org_id**: если метод принимает `orgID`, он фигурирует в тесте (передаётся в вызов, проверяется в ассертах).

Порог: **baseline обычно ≤ 2/5 в среднем, tuned цель ≥ 4.5/5 в среднем.**

## Блок 2. Структурное сходство с эталоном

Проверяем попарно `baseline_output` vs `reference` и `tuned_output` vs `reference`:

1. **Форма теста совпадает**: table-driven (`tests := []struct{...}{}` + `for _, tt := range tests`) vs plain — сравниваем одно с одним.
2. **Количество кейсов в пределах ±1** от эталона (если эталон табличный).
3. **Используемые поля структур совпадают** (по списку ассертов / сетапа): модель не выдумывает несуществующие поля типа `db.Campaign`.

Порог: **baseline часто промахивается на п.1 и п.3, tuned должен попадать 9/10.**

## Блок 3. Project-specific конвенции (самое ценное)

Это то, чему base-модель **не обучена в принципе** и где файнтюн даст очевидный выигрыш:

1. **Мок-паттерн**: использует `&mockDBTX{queryRowFn: ...}` или `{queryFn: ...}` — не пишет свой `gomock`/`testify/mock`.
2. **Error-assert паттерн**: `if err != model.ErrNotFound` / `if err == nil { t.Fatal(...) }` — не тянет `assert.ErrorIs` или `require`.
3. **Импорты не выдуманы**: модель не вставляет `import "testify/assert"` или левых `require`-пакетов (в нашем выходе импортов быть не должно вообще — блок 1 пункт 3).
4. **Комментарий-документация над тестом**: короткий `// TestXxx verifies ...` строкой выше функции — факультативно, но присутствует у большинства эталонов.

Порог: **baseline обычно провалит п.1 и п.2 во всех DB-error кейсах (5 из 10). Tuned должен попадать ≥ 9/10.**

## Чего НЕ проверяем (сознательно)

- Семантическая корректность ассертов (типа «правильно ли проверяется значение X») — для этого нужен ручной code review на каждый кейс, не масштабируется на 10 примеров.
- Стиль комментариев на уровне формулировок.
- Производительность теста.

Эти вещи оцениваются выборочно в видео-защите на 2-3 примерах.
